<objective>
Create the static content layer that makes `minilegion context <tool>` produce rich, tool-specific output: adapter definition files, per-stage prompt templates, memory scaffolding, and the ADR-0007 record.

Also wire `minilegion init` to scaffold these directories automatically so new projects start with the full structure.

Purpose: Without adapters and templates, the context command assembles only raw state data. This plan adds the framing layer — the adapter tells the AI tool how to interpret the context; the template tells it what to do next; the memory files carry forward decisions and constraints across sessions.

Output:
- `project-ai/adapters/` — 5 files (_base + 4 tools)
- `project-ai/templates/` — 8 stage templates (one per Stage)
- `project-ai/memory/` — 3 scaffold files
- `minilegion init` creates all of the above
- ADR-0007 verified present at `.planning/milestones/v1.1-ADR-0007.md`
- Integration tests: `assemble_context` uses template + adapter when files exist
</objective>

**Part A — Define adapter and template content as module-level constants in `commands.py`** (alongside the existing `BRIEF_TEMPLATE` constant). Do not use external files bundled with the package — keep content in source for simplicity.

Adapter content (write these strings):

`ADAPTER_BASE` — Generic instructions, ~10 lines:
```
# MiniLegion Context Block

You are resuming work on a MiniLegion project. The context block below was assembled
by `minilegion context` and describes the current project state.

## How to use this context
1. Read the **Current State** section to understand where the pipeline is.
2. Read the **Previous Artifact** section for the most recent output.
3. Read the **Stage Template** section for guidance on what to produce next.
4. Check **Memory** for decisions and constraints that must be respected.
5. Proceed based on this context — do not ask the user to re-explain the project.
```

`ADAPTER_CLAUDE` — Extends base with Claude-specific framing:
```
# Claude — MiniLegion Context

<!-- Paste this at the start of a new Claude conversation. -->

You are a senior software engineer resuming a MiniLegion-orchestrated project.
The structured context below is authoritative — trust it over your prior knowledge of the codebase.
Follow the stage template exactly. Respect all constraints in the Memory section.
```

`ADAPTER_CHATGPT` — ChatGPT framing:
```
# ChatGPT — MiniLegion Context

<!-- Paste this at the start of a new ChatGPT conversation. -->

You are resuming a software project managed by MiniLegion. Use the structured context
below as your single source of truth. Follow the current stage template and respect all
constraints listed in the Memory section.
```

`ADAPTER_COPILOT` — Copilot Chat framing:
```
# GitHub Copilot — MiniLegion Context

<!-- Paste this into a Copilot Chat conversation in VS Code or GitHub.com. -->

Use the project context below to assist with the current pipeline stage.
Do not suggest changes outside the scope defined in the Stage Template.
```

`ADAPTER_OPENCODE` — OpenCode framing:
```
# OpenCode — MiniLegion Context

<!-- Paste this as the initial message in an OpenCode session. -->

You are resuming a MiniLegion project. The context block below is authoritative.
Execute the task described in the Stage Template section. Follow all constraints in Memory.
Produce output in the format expected by MiniLegion's schema validation.
```

Stage templates — one per `Stage` enum value. Each template is a short markdown doc (~5-8 lines) that describes what the AI should produce at that stage. Key stages:

- `init.md`: "No active pipeline stage. Run `minilegion brief <text>` to start."
- `brief.md`: "Review the project brief. Ensure it clearly states goals, constraints, and success criteria."
- `research.md`: "Analyze the codebase and brief. Produce RESEARCH.json with: project_overview, tech_stack, existing_conventions, open_questions, codebase_insights."
- `design.md`: "Based on research, produce DESIGN.json with: design_approach, architecture_decisions, components, data_models, api_contracts, integration_points, design_patterns_used, conventions_to_follow, technical_risks, out_of_scope, test_strategy, estimated_complexity."
- `plan.md`: "Based on design, produce PLAN.json with: summary, tasks (each with id, description, type, action, touched_files), total_tasks, estimated_complexity."
- `execute.md`: "Implement the plan tasks. Produce EXECUTION_LOG.json with: tasks (each with task_id, status, changed_files, notes), summary."
- `review.md`: "Review the execution log against the plan and design. Produce REVIEW.json with: verdict (pass/revise), summary, issues_found, corrective_actions, design_conformity."
- `archive.md`: "Pipeline complete. Run `minilegion archive` to finalize. No LLM output required."

Memory scaffolds:
- `decisions.md`: `# Decisions\n\n_Add key project decisions here. Each entry: date, decision, rationale._\n`
- `glossary.md`: `# Glossary\n\n_Add domain terms here. Each entry: term — definition._\n`
- `constraints.md`: `# Constraints\n\n_Add hard constraints here (technical, business, security). Each entry: constraint — why it applies._\n`
